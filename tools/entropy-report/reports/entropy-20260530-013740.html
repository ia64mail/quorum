First report on the corrected lexer (ticket #50). This is the QRM8 milestone report regenerated after the Halstead-correctness pass — template-literal interpolations are now tokenized rather than swallowed, regex literals count as single operands, string quote styles and numeric separators are normalized, per-app Volume is computed from per-app union vocabularies, and Estimated Bugs uses Halstead's canonical B = E^(2/3) / 3000. The absolute figures here are therefore not directly comparable to earlier reports — especially the per-app Volumes, which rise sharply under the union-map method. Read this report as the new, more faithful baseline rather than a continuation of the previous numeric series.

Overall trajectory — sub-linear and stabilizing, punctuated by deliberate contractions. Across 534 commits the codebase reached a Halstead Volume of 1,385,067 over 17,518 LOC in 147 files. Growth is decidedly not exponential: Volume climbed steeply through QRM1–QRM2 (first source commit ≈ 10,170 → ~444,000 by commit #41) as the NestJS/MCP scaffolding landed, then entered a long, near-linear accrual. The two largest single-commit swings are both removals — a4071b0 "QRM6-009: remove legacy terminal app" (Volume Δ−228,317) and the QRM8 roadmap relocation off the source tree (Δ−192,921). The terminal-app removal is the clearest inflection point: aggregate Volume fell from a QRM5-era peak of 1,254,846 (#291) to 1,035,805 (#301), after which QRM6→QRM8 rebuilt to today's 1.39M through steady feature work rather than a spike. This is the entropy signature of a codebase under active curation, not unmanaged sprawl.

Volume and Difficulty have decoupled — the code is getting larger, not harder. Halstead Difficulty has been remarkably flat for the entire mature life of the project: it reached 339.8 at commit #41 and sits at 391.0 today — roughly +15% over ~490 commits during which Volume more than tripled (444k → 1.39M, +212%). Difficulty oscillates in a tight ~340–392 band even as information content triples. Because aggregate D is driven by operand-vocabulary reuse (η₁/2 × N₂/η₂), this flatness means new code is built largely from the existing vocabulary of operators and identifiers — agents extend established patterns (NestJS modules, MCP tool handlers, Zod schemas, the agentic tool loop) rather than inventing novel constructs. The marginal cost of comprehension per unit of new code is roughly constant.

Quality assessment — treat the absolute numbers as trend signals, not verdicts. The corrected Estimated-Bugs predictor (E^(2/3)/3000) reports 221.5 latent bugs — about half the inflated 455 the old V/3000 variant produced — and aggregate Difficulty of 391 looks alarming against the D=10–30 typical of an individual well-factored TypeScript file. Both remain artifacts of aggregation: Halstead was designed for single modules, and folding a 147-file monorepo into one vocabulary inflates the figures super-linearly. The meaningful reading is the trajectory (flat Difficulty, sub-linear Volume) and its divergence from ground truth — the QRM4→QRM8 release notes record 0 bugs in new feature code and a 0% post-review fix rate across five consecutive milestones. A predictor calibrated on human-authored code projecting hundreds of bugs against an observed zero in reviewed feature code is itself a data point: ticket-scoped, code-reviewed AI generation produces lower realized defect density than the information-theoretic proxy assumes.

Per-app distribution — concentrated in the MCP server. Under the corrected union-map method each app's Volume is computed from its own vocabulary, so the figures are larger than before and do not sum to the project total (1.39M) — the apps share substantial vocabulary. mcp-server carries 736,049 Volume (75 files, 9,815 LOC), agent holds 418,588 (40 files, 5,754 LOC), and common stays lean at 107,574 (32 files, 1,949 LOC); the legacy terminal app is gone. The server-heavy profile is appropriate — it hosts the 7 tools, 2 resources, Agent Registry, Message Broker, and Context Store — and the lean common library confirms QRM6–QRM8 growth was additive at the handler layer rather than a churn of shared abstractions.

What this reveals about AI-generated entropy, and recommendations. Three milestones of evidence (QRM6's net-negative refactor, QRM7's transport stabilization, QRM8's isolation layer) show AI-authored code under ticket-level control and code review does not exhibit the runaway entropy often feared of "vibe coding": Difficulty is flat, the largest deltas are intentional deletions, and the system is willing to remove code (the terminal app, −228k) when an abstraction is retired. Recommendations: (1) report the aggregate Difficulty and Estimated-Bugs figures as trend lines, never as bug counts — pair them with the realized 0-defect record to avoid false alarm; (2) run a follow-up cyclomatic pass with lizard on the agent app's InvocationHandler commit/push and worktree-lifecycle paths, since Halstead's flat aggregate Difficulty cannot detect a single deeply-branched handler — exactly the kind of logic QRM8 added; (3) continue watching Volume-per-file (now ≈ 9,422) as the truest early-warning signal for bloat, since it normalizes away raw growth.
